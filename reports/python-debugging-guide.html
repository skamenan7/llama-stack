Zero to Hero Guide for Java Developers

1. Java vs Python Debugging Concepts

Good news: debugging concepts are universal. Your Java debugging skills transfer directly!

Concept Java (IntelliJ/Eclipse) Python (Cursor/PyCharm)
Debugger Protocol JDWP (Java Debug Wire Protocol) DAP (Debug Adapter Protocol) via debugpy
Breakpoint Click in gutter Same - click in gutter
Step Over F8 F10 (Cursor) / F8 (PyCharm)
Step Into F7 F11 (Cursor) / F7 (PyCharm)
Step Out Shift+F8 Shift+F11 (Cursor) / Shift+F8 (PyCharm)
Continue F9 F5 (Cursor) / F9 (PyCharm)
Watch Variables Variables panel Same - Variables panel
Evaluate Expression Alt+F8 Debug Console / Alt+F8
Conditional Breakpoint Right-click breakpoint Same - right-click breakpoint
The main difference: Python uses debugpy as the debug adapter, while Java uses JDWP. But the user experience is nearly identical. If you can debug Java, you can debug Python!

2. The Debugging Mental Model

When you click "Send Request" in your HTML form, here's the full flow:

sequenceDiagram participant Browser as Browser/HTML participant Server as Python Server participant Route as Route Handler participant Logic as Business Logic participant DB as Database/API Browser->>Server: HTTP Request (POST /api/chat) Note over Server: Request hits ASGI/WSGI Server->>Route: Route matched Note over Route: YOUR BREAKPOINT HERE Route->>Logic: Call service function Logic->>DB: Query/External call DB-->>Logic: Response data Logic-->>Route: Return result Route-->>Server: HTTP Response Server-->>Browser: JSON Response

Key Insight for Server Debugging

Unlike Java Spring where you might debug from main(), Python web servers (FastAPI, Flask) are:

  • Event-driven - code runs when requests arrive
  • Often async - uses async/await (like CompletableFuture)
  • Hot-reloadable - changes apply without restart (usually)
You don't need to trace from if __name__ == "__main__". Just set a breakpoint in your route handler and send a request!

3. Setup: Cursor IDE (VS Code)

Step 1: Create launch.json

Open Run and Debug Panel

Press Ctrl+Shift+D (or click the bug icon in sidebar)

Create launch.json

Click "create a launch.json file" > Select "Python Debugger"

Configure for Your Server

Replace the contents with one of these configurations:

For FastAPI (uvicorn):

{
    "version": "0.2.0",
    "configurations": [
        {
            "name": "FastAPI Debug",
            "type": "debugpy",
            "request": "launch",
            "module": "uvicorn",
            "args": [
                "llama_stack.distribution.server.server:app",
                "--host", "0.0.0.0",
                "--port", "8321",
                "--reload"
            ],
            "jinja": true,
            "env": {
                "PYTHONPATH": "${workspaceFolder}"
            },
            "console": "integratedTerminal"
        }
    ]
}

For Flask:

{
    "version": "0.2.0",
    "configurations": [
        {
            "name": "Flask Debug",
            "type": "debugpy",
            "request": "launch",
            "module": "flask",
            "args": ["run", "--debug", "--port", "5000"],
            "env": {
                "FLASK_APP": "app.py",
                "PYTHONPATH": "${workspaceFolder}"
            },
            "console": "integratedTerminal"
        }
    ]
}

For generic Python script:

{
    "version": "0.2.0",
    "configurations": [
        {
            "name": "Python: Current File",
            "type": "debugpy",
            "request": "launch",
            "program": "${file}",
            "console": "integratedTerminal"
        }
    ]
}
Common Mistake: Using "type": "python" instead of "type": "debugpy". The old Python extension used "python", but the new debugger requires "debugpy".

Step 2: Set Breakpoints

flowchart LR A["Click line number gutter"] --> B["Red dot appears"] B --> C["Run debugger F5"] C --> D["Send HTTP request"] D --> E["Execution pauses at breakpoint"] E --> F["Inspect variables"]

Click in the gutter (left of line numbers) to set a breakpoint. A red dot appears.

from fastapi import FastAPI, Request

app = FastAPI()

@app.post("/api/chat")
async def chat_endpoint(request: Request):
    body = await request.json()  # <-- Set breakpoint here!

    message = body.get("message")  # <-- Or here to see parsed body

    # Your business logic
    response = process_message(message)

    return {"response": response}

Step 3: Start Debugging

Press F5 or click green play button

The server starts in debug mode. You'll see the debug toolbar appear.

Send your request

Open your HTML page and click "Send Request". The debugger pauses at your breakpoint.

Explore!

Use the debug panel to inspect variables, step through code, and understand the flow.

Remote Debugging (Attach to Running Process)

If your server is already running (e.g., in Docker), you can attach the debugger:

Install debugpy in your environment

pip install debugpy

Add debug listener to your server

# At the top of your server file
import debugpy
debugpy.listen(("0.0.0.0", 5678))
print("Waiting for debugger attach...")
debugpy.wait_for_client()  # Optional: pause until debugger connects

Create Remote Debug config in PyCharm

Run > Edit Configurations > + > Python Debug Server

  • Host: localhost
  • Port: 5678

6. Breakpoint Types

Beyond simple breakpoints, both IDEs support advanced breakpoint features:

Line Breakpoint (Basic)

Click in gutter. Stops every time that line executes.

Conditional Breakpoint

Right-click the breakpoint > Add condition. Only stops when condition is true.

# Right-click breakpoint, add condition: message == "debug"
@app.post("/api/chat")
async def chat(request: Request):
    body = await request.json()
    message = body.get("message")  # <-- Conditional breakpoint here
    # Only stops if message == "debug"
In Java, you'd right-click and add a condition in IntelliJ. Identical experience in Python IDEs!

Logpoint (Non-breaking)

Right-click > Logpoint. Logs a message without stopping execution.

# Logpoint message: "Processing message: {message}"
# This prints to debug console without pausing

Exception Breakpoint

Stops when specific exceptions are raised.

In Debug panel > Breakpoints section > Click "+" > "Exception Breakpoint"

Or: Run and Debug > "Raised Exceptions" toggle

Run > View Breakpoints (Ctrl+Shift+F8)

Click "+" > "Python Exception Breakpoint" > Enter exception class name

Function Breakpoint

Break when a specific function is called (by name, without finding the code).

Variables Panel

Shows all variables in current scope. Click to expand objects/dicts.

You can modify variable values while debugging! Double-click a value in the Variables panel to edit it.

Watch Expressions

Add expressions to monitor across steps. Example watches:

  • len(messages)
  • request.headers.get("Authorization")
  • user.is_authenticated

Call Stack

Shows the function call chain. Click any frame to see its local variables.

Just like IntelliJ's "Frames" panel. Click different stack frames to navigate up/down the call hierarchy.

Debug Console

Interactive Python REPL at the breakpoint! Type any expression:

# In debug console:
>>> body

>>> import json; json.dumps(body, indent=2)

8. Server Request Debugging

Here's the complete flow for debugging your server when a request comes in:

The Complete Picture

sequenceDiagram participant HTML as Your HTML Page participant IDE as Cursor/PyCharm participant Server as Python Server participant Code as Your Route Handler Note over IDE: Start debugger (F5) IDE->>Server: Launch server in debug mode Server-->>IDE: Debugpy connected Note over HTML: Click "Send Request" HTML->>Server: POST /api/endpoint Server->>Code: Route matched, call handler Note over Code: BREAKPOINT HIT Code-->>IDE: Execution paused Note over IDE: You inspect variables,
step through code... IDE->>Code: Continue (F5) Code-->>Server: Return response Server-->>HTML: JSON response displayed

Where to Set Breakpoints

llama_stack/ distribution/ server/ server.py # Main FastAPI app - good for middleware providers/ impls/ inference/ ollama/ # Provider implementation - set breakpoints here! apis/ inference/ # API definitions - route handlers

Example: Debug a Chat Request

# In your route handler (e.g., inference route)
@router.post("/inference/chat_completion")
async def chat_completion(
    request: ChatCompletionRequest,
    # ...
):
    # BREAKPOINT 1: See the incoming request
    logger.info(f"Received request: {request}")

    # BREAKPOINT 2: Before calling the provider
    provider = self.get_provider(request.model)

    # BREAKPOINT 3: See provider response
    response = await provider.chat_completion(request)

    return response  # BREAKPOINT 4: See final response
Hot Reload Gotcha: With --reload, the server restarts on file changes. Your debugger may disconnect. Use "justMyCode": false in launch.json if you need to debug library code too.

9. Async/Await Debugging

Python's async/await is similar to Java's CompletableFuture. The debugger handles it seamlessly.

Think of await like .get() on a Future. When you step over an await, the debugger waits for the async operation to complete.

Async Call Stack

The call stack shows async frames. You'll see the full chain even across await boundaries.

async def handler():
    result = await fetch_data()  # Stepping over this waits for completion
    return result

async def fetch_data():
    async with aiohttp.ClientSession() as session:
        response = await session.get(url)  # Breakpoint here shows full stack
        return await response.json()

Debugging Concurrent Tasks

When debugging asyncio.gather() or multiple tasks:

async def parallel_requests():
    # All three run concurrently
    results = await asyncio.gather(
        fetch_users(),    # Task 1
        fetch_orders(),   # Task 2
        fetch_products()  # Task 3
    )
    return results  # Breakpoint here: all tasks completed
Set breakpoints in each async function. The debugger will stop at whichever completes first, letting you see the order of execution.

10. Advanced Debugging Techniques

1. Post-Mortem Debugging

Debug after an exception occurred:

import pdb

try:
    risky_operation()
except Exception:
    pdb.post_mortem()  # Drops into debugger at exception point

2. Remote Debugging

Debug code running in Docker or on a remote server:

# In your server code:
import debugpy
debugpy.listen(("0.0.0.0", 5678))
debugpy.wait_for_client()  # Server pauses until debugger attaches
// In launch.json (Cursor)
{
    "name": "Attach to Remote",
    "type": "debugpy",
    "request": "attach",
    "connect": {
        "host": "localhost",
        "port": 5678
    }
}

12. Troubleshooting Common Issues

Debugger Won't Start

Problem: "Unable to import debugpy"
Solution: pip install debugpy in your active environment

Breakpoints Not Hit

Causes:
  • Wrong Python interpreter selected
  • Code path not executed (check your request)
  • Hot reload created new process (restart debugger)
  • Breakpoint in imported module not loaded yet
Solution: Check interpreter in status bar, verify route is correct

Variables Show "<not available>"

Cause: Optimized code or variable out of scope
Solution: Set PYTHONDONTWRITEBYTECODE=1 or disable optimization

Server Won't Stop

Cause: Uvicorn/process not terminating cleanly
Solution: Kill manually: pkill -f uvicorn or use Ctrl+C multiple times

Timeout During Debugging

Cause: HTTP client times out while you're stepping through code
Solution: Increase timeout in your HTTP client or use curl with no timeout
Pro tip: Use curl from terminal for requests during debugging - it's more patient than browsers! 
curl -X POST http://localhost:8321/api/chat \
  -H "Content-Type: application/json" \
  -d '{"message": "test"}' \
  --max-time 300
